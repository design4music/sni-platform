# SEO Phase 2 — Document 3: Sitemap & Indexability Diagnosis

**Date**: 2026-05-12
**Live URLs**: `sitemap-index.xml`, `sitemap.xml`, `sitemap-days.xml`
**Background**: User reports that Google Search Console shows "Couldn't fetch"
on submitted sitemaps despite them having been added ~2 weeks ago. ~9,000 pages
currently indexed.

## TL;DR

The sitemap files **work and serve fast from Googlebot's UA** today. Three
material defects remain that almost certainly cause GSC "Couldn't fetch" / soft
errors:

1. **`sitemap.xml` ships `Content-Type: application/xml` without a `charset` parameter** while the other two sitemap routes ship `application/xml; charset=utf-8`. RFC-strict XML parsers (Google's included) treat missing charset as ambiguity for non-ASCII content.
2. **`sitemap.xml` ships `Cache-Control: public, max-age=0, must-revalidate`** (the Next.js default for `revalidate = false` on `app/sitemap.ts`). Googlebot fetches will hit origin every time and may time out at scale. Other sitemap routes ship `max-age=3600, s-maxage=86400`.
3. **`sitemap.xml` is missing the `hreflang="en"` self-reference** on every `<url>` entry. Each URL lists only `de` and `x-default` alternates. Per Google's reciprocal-cluster rule, every URL in a language set must list itself. The bug originates in the `alt()` helper in `app/sitemap.ts`. `sitemap-days.xml` does this correctly because it serializes the XML by hand.

Beyond those, two smaller issues:

4. The sitemap loop emits **3 dead `/signals/*` URLs** (`commodities`, `policies`, `systems`) that return HTTP 200 with thin / soft-404 content. ×2 locales = 6 soft-404s for Google to clean up.
5. **`/c/[centroid_key]/about` (75 pages × 2 locales = 150 URLs)** are not included in any sitemap.

## Defect details

### S-1 — `sitemap.xml` missing `charset=utf-8` in Content-Type

Next.js's default `Metadata­Route.Sitemap` handler emits `Content-Type:
application/xml` without the charset parameter. The two custom routes
(`sitemap-index.xml/route.ts`, `sitemap-days.xml/route.ts`) explicitly set
`application/xml; charset=utf-8` in their `new Response(body, { headers: … })`
calls.

Google says it can parse either, but the sitemap spec recommends UTF-8 and many
diagnostic tools (and probably GSC's internal validator) flag missing-charset
responses as "Couldn't fetch" or "Wrong content-type" even when the body is
valid UTF-8.

**Fix**: convert `app/sitemap.ts` to a custom Route Handler at
`app/sitemap.xml/route.ts` returning a `new Response(body, { headers: …})` with
both charset and cache-control set correctly. The body construction can call
the same data queries; the wrap is what's missing.

(Alternative: keep `app/sitemap.ts` and add `headers()` middleware to inject
the charset — but a Route Handler is the more idiomatic fix for the cache
issue below as well.)

### S-2 — `sitemap.xml` uncached (`max-age=0, must-revalidate`)

`app/sitemap.ts` has `export const revalidate = false;`. Next.js renders the
sitemap on demand and serves it with the default no-cache header. The
`x-nextjs-cache: HIT` shows Cloudflare is doing its own edge caching anyway,
but the response header tells Googlebot "always re-fetch", which compounds
with S-1 to produce timeouts at scale (the body construction does ~12 DB
queries; a cold cache fetch will spend 2–3 s).

The two custom sitemaps set `s-maxage=86400` and serve identical content for
24 h, regenerated by `.github/workflows/revalidate-sitemap.yml` calling
`/api/cron/revalidate-sitemap` daily.

**Fix**: bundled with S-1. The Route Handler form lets us set the same
`s-maxage=86400` header.

### S-3 — Missing `hreflang="en"` self-reference (every URL entry in `sitemap.xml`)

Sample entry from production:

```xml
<url>
<loc>https://www.worldbrief.info/trending</loc>
<xhtml:link rel="alternate" hreflang="de" href="https://www.worldbrief.info/de/trending" />
<xhtml:link rel="alternate" hreflang="x-default" href="https://www.worldbrief.info/trending" />
<changefreq>daily</changefreq>
<priority>0.9</priority>
</url>
```

Per Google: *"Each language version must list itself and all other versions"*.
The `en` self-reference is required for the cluster to be valid; without it,
Google may report "Sitemap could not be read" or "Unexpected hreflang" issues
under Coverage / International Targeting.

Root cause is the `alt()` helper at `app/sitemap.ts:27-34`:

```ts
function alt(path: string) {
  return {
    languages: {
      de: `${SITE_URL}/de${path}`,
      'x-default': `${SITE_URL}${path}`,
    },
  };
}
```

Add an `en` key:

```ts
en: `${SITE_URL}${path}`,
```

`sitemap-days.xml/route.ts` already does this correctly by writing the XML by
hand.

### S-4 — Dead signal-category URLs in sitemap

`app/sitemap.ts:189` hardcodes 7 signal-type slugs:

```ts
const signalTypes = ['persons','orgs','places','commodities','policies','systems','named_events'];
```

But `app/[locale]/signals/[type]/page.tsx:19-21` accepts only 4:

```ts
const VALID_TYPES = new Set<SignalType>([
  'persons','orgs','places','named_events',
]);
```

The other 3 (`commodities`, `policies`, `systems`) return `notFound()` from the
page handler. Live verification: all three respond HTTP 200 with a thin
"Signal Category Not Found" body (Next.js renders the 404 view inline) — i.e.
soft-404. ×2 locales = 6 URLs Google wastes crawl budget on, plus likely a
warning in GSC Coverage report.

**Fix**: trim the sitemap list to `['persons','orgs','places','named_events']`.
One-line change.

### S-5 — `/c/[centroid_key]/about` missing from sitemap

`apps/frontend/app/[locale]/c/[centroid_key]/about/page.tsx` ships full
metadata (canonical, hreflang, OG-article, description from
`profile_json.tldr`). The sitemap loop at `app/sitemap.ts:89-107` only emits
the centroid and per-track URLs, never the `/about` sibling.

75 centroids × 2 locales = 150 URLs of high-quality curated background. They
already exist and ship correctly — just need to be advertised.

**Fix**: in the centroid block of the sitemap loop, add:

```ts
entries.push({
  url: `${SITE_URL}/c/${centroidId}/about`,
  changeFrequency: 'monthly',
  priority: 0.6,
  alternates: alt(`/c/${centroidId}/about`),
});
```

## Hreflang / canonical consistency on rendered pages

Per-page heads were verified in Document 2 and look good across the in-scope
templates. Only the **`/signals/*` family** is missing canonical + hreflang
entirely (those routes don't go through `buildPageMetadata`).

Important pairing: even after fixing the sitemap hreflang (S-3), Google still
needs the same hreflang information on the rendered page. Track-day, centroid,
event, etc. all have it. Adding `buildPageMetadata` to `/signals/[type]`,
`/signals/[type]/[value]`, and `/signals` closes the only remaining hole.

## GSC submission status — what to do next

Based on the diagnoses above, the recommended sequence:

1. **Land S-1 + S-2 + S-3 together** (single Route-Handler rewrite of
   `sitemap.xml`). This addresses the three issues most likely to be triggering
   "Couldn't fetch".
2. Land **S-4** and **S-5** in the same commit (tiny code edits inside the
   same file).
3. Cron-trigger `/api/cron/revalidate-sitemap` to regenerate the cached body.
4. In GSC, **remove and re-add** `https://www.worldbrief.info/sitemap-index.xml`.
   (Removing then re-adding forces a fresh fetch attempt; GSC caches its own
   "Couldn't fetch" verdict for a long time.)
5. Wait 24–72 h for GSC to re-process. The two underlying sitemaps will be
   re-fetched as part of index processing.
6. Verify in Coverage report:
   - Sitemap status = "Success"
   - "Indexed, not submitted in sitemap" count should drop as `/about` pages get listed
   - Soft-404 count should drop as 3 dead `/signals/*` URLs are removed
   - International Targeting tab should clear any "Missing return tag" hreflang issues
